# Product Standards & Engineering Principles

> 作者：Lumoryx · 版本：v1.1 · 日期：2026-05-11
>
> 本文档定义了所有产品从立项到交付的强制规范和最佳实践。
> 每一条规范背后都有明确的"为什么"，不是约束，是护城河。

---

## 〇、核心信条

> **需求驱动一切。设计先于代码。规范创造速度。**

任何产品决策，先问三个问题：
1. **用户为什么需要它？**（需求合理性）
2. **用户如何感知它？**（体验设计）
3. **工程如何实现它？**（技术方案）

顺序不可颠倒。

## 一、研发流程规范

### 1.1 标准研发阶段（强制执行，不可跳步）

```
阶段 1 — 产品需求文档 (PRD)
    ↓  评审通过后
阶段 2 — UI/UX 设计稿
    ↓  评审通过后
阶段 3 — 技术方案设计 (TDD)
    ↓  评审通过后
阶段 4 — 开发 & Code Review
    ↓
阶段 5 — 测试 & 上线
```

**绝对禁止**：需求未明确就开始写代码。发现需求不清晰，立刻停下来补 PRD，不要"边做边想"。

### 1.2 PRD 必须包含的内容

每份 PRD 必须覆盖以下章节，缺项不得进入下一阶段：

| 章节 | 内容要求 |
|------|---------|
| 产品定位 | 一句话定位（电梯演讲级别）+ 竞品差异化分析 |
| 目标用户 | 用户画像 + 核心痛点排序（≥3条，按严重程度） |
| 功能范围 | 本期 P0/P1/P2 功能列表 + 明确的 out-of-scope |
| 用户流程 | 核心路径流程图（文字或 ASCII 图均可） |
| 成功指标 | 北极星指标 + 至少 3 个可量化的支撑指标（KPI） |
| 数据模型 | 核心数据表结构（字段 + 类型 + 关系） |
| 非功能需求 | 多语言、多主题、安全、性能要求 |

### 1.3 需求变更规范

- **小改动**（不影响架构）：更新 PRD 版本号（patch），注明变更历史，直接执行
- **中改动**（影响数据模型或流程）：更新 PRD 版本号（minor），重新评审技术方案
- **大改动**（颠覆核心功能）：重新立项，PRD 版本号 major

## 二、产品设计强制规范

### 2.1 多语言支持（强制）

**所有面向用户的产品，必须从第一天支持多语言。**

#### 最低要求
- Phase 1 必须支持：**简体中文（zh_CN）** + **English（en_US）**
- 界面语言跟随系统语言自动切换，用户可手动覆盖
- 语言偏好持久化存储到用户配置

#### 实现标准
- **Flutter**：使用 `flutter_localizations` + `intl`，所有字符串写入 `.arb` 文件
- **React/Web**：使用 `i18next` 或 `react-intl`，所有字符串写入 JSON locale 文件
- **禁止**：在代码中硬编码任何用户可见的文字（中文或英文均不允许）
- **命名规范**：Key 格式为 `{feature}_{component}_{description}`，例如 `home_greeting_morning`

#### 语言扩展路径（推荐顺序）
```
Phase 1: zh_CN + en_US
Phase 2: zh_TW（繁体中文）
Phase 3: ja_JP / ko_KR（东亚市场）
Phase 4: es_ES / fr_FR / de_DE（欧洲市场）
```

### 2.2 多主题色彩系统（强制）

**所有面向用户的产品，必须支持至少 2 套主题色 + Light/Dark 模式切换。**

#### 最低要求
- 预设主题 ≥ 2 套（推荐 4 套，覆盖冷暖色系）
- 每套主题必须包含完整的 Light Mode + Dark Mode
- 外观模式：跟随系统 / 强制浅色 / 强制深色（三选一）
- 主题偏好持久化存储到用户配置

#### 色彩 Token 规范（所有产品强制采用语义化 Token）

禁止在组件中硬编码颜色值。所有颜色必须通过语义化 Token 引用：

```
必须定义的 Token（最小集）：
  primary           — 主色（按钮、进度条、高亮）
  primary_variant   — 主色变体（渐变、hover 态）
  background        — 页面底色
  surface           — 卡片/面板背景
  on_primary        — 主色上的文字
  on_surface        — 卡片上的主文字
  on_surface_muted  — 次要文字（建议 60% 透明度）
  success           — 成功态（绿系）
  error             — 错误态（红系）
  divider           — 分割线（低透明度）
```

#### 主题切换实现
- **Flutter**：`ThemeExtension<T>` + Riverpod `StateProvider`，热切换无需重启
- **React**：CSS Variables + Context Provider，切换只更新 CSS 变量

#### 进阶：里程碑解锁主题
核心留存机制之一：通过用户行为（连续打卡、达成成就等）解锁限定主题，在设置页展示锁定状态，激励用户持续使用。

### 2.3 UI 设计哲学（强制）

**目标**：高端 · 克制 · 沉浸。让用户感觉在用一个"贵"的产品。

#### 五条铁律

| 铁律 | 说明 |
|------|------|
| **5秒原则** | 任何页面，用户 5 秒内必须知道该做什么。否则重新设计 |
| **进度永远可见** | 用户任何时候都知道自己在哪、还剩多少，不能让用户迷失 |
| **留白即设计** | 高端感来自克制，不堆砌元素。三个卡片不如一个卡片做好 |
| **庆祝每个小胜利** | 完成一个小步骤就要有即时反馈，不只庆祝最终完成 |
| **微交互丰富** | 按钮弹性、卡片翻转、数字跳动。动效是产品温度的来源 |

#### 动效规范

| 场景 | 推荐时长 | 曲线 |
|------|---------|------|
| 页面切换 | 250-350ms | ease-in-out |
| 卡片展开/翻转 | 350-450ms | ease-in-out |
| 庆祝动画 | 1500-2500ms | Lottie |
| 按钮按压反馈 | 100-150ms | ease-out |
| 数字跳动 | 400-600ms | spring/bounce |
| 进度条填充 | 600-900ms | ease-out |

#### 字体规范

- **英文展示字体**（大标题、关键数字）：使用衬线字体增加仪式感（Playfair Display / SF Pro Display）
- **正文字体**：系统字体（SF Pro / Roboto / Inter）
- **行高**：正文固定 1.6-1.7，不低于 1.5

### 2.4 空状态设计（强制）

每个列表页、数据页必须设计 Empty State，不允许出现空白页面。

空状态必须包含：
1. 说明性插图或图标
2. 一句话解释"为什么是空的"
3. 明确的行动按钮（引导用户做下一步）

## 三、习惯化产品设计规范

适用于所有以"养成用户使用习惯"为目标的产品（学习类、健康类、效率类）。

### 3.1 习惯循环设计（Habit Loop）

每个核心功能必须设计完整的四步循环：

```
TRIGGER（触发）→ ROUTINE（行动）→ REWARD（奖励）→ INVESTMENT（投入）
     ↑_______________________________________________|
```

| 环节 | 设计要点 |
|------|---------|
| **触发** | 智能推送（学习最佳时间）+ 损失提醒（Streak 即将中断）+ 社交触发（好友完成了） |
| **行动** | 严格控制完成时间（5-10分钟），永远显示剩余步骤，所有交互尽量单手操作 |
| **奖励** | 即时反馈（动画庆祝）+ 里程碑奖励（实物/虚拟权益）+ 可变奖励（惊喜感） |
| **投入** | 积累感（数据/内容/社交关系）让用户舍不得离开 |

### 3.2 Streak 系统设计标准

Streak（连续打卡）是习惯类产品留存的核心机制，必须精心设计：

- **显示位置**：首页最醒目位置，数字+火焰图标，颜色专属
- **断裂保护**：提供冻结卡机制（付费用户专享），防止因偶发原因导致 Streak 清零引发流失
- **里程碑节点**：7天、30天、100天设立明确奖励，提前 1-3 天推送预告
- **情感化**：Streak 数字字体要大，颜色要暖，让用户产生"数字依赖"
- **Streak 焦虑平衡**：不能让 Streak 成为压力来源，设计上要有温度（允许 1 天弹性 / 冻结卡）

### 3.3 推送通知规范

推送是留存的杠杆，但烂推送是卸载的原因。

**四不原则**：
- 不在非活跃时段推（22:00 - 08:00 默认静默）
- 不推无价值信息（促销优先级最低）
- 不高频推送（同类通知每日最多 1 条）
- 不推用户已完成的任务

**文案标准**：
- 有具体内容（"今日单词：ephemeral"）比泛泛更好
- 制造轻微紧迫感但不焦虑（"再 3 天就免单了"）
- 口语化、有个性，不要像系统通知

## 四、商业模式规范

### 4.1 订阅制产品标准

所有订阅制产品必须遵循以下设计：

#### 试用设计
- 新用户必须提供免费试用期（推荐 7-14 天）
- 试用期内提供完整 Pro 功能体验（不要阉割版）
- 不建议试用要求绑卡（降低注册门槛）
- 试用结束前 3 天发送提醒，附上用户数据报告（用数据证明价值）

#### 定价策略
- 必须提供月付 + 年付两个选项，年付展示节省比例
- 年付默认选中（提高年付比例，降低流失率）
- 价格锚定：年付价格 ≤ 月付 × 7（至少省 5 个月）

#### 忠诚奖励机制（Loyalty Reward）
长期付费用户必须设计忠诚奖励，将"坚持使用"与"实际权益"挂钩：

- 标准方案：连续使用满 N 天 → 免下期账单
- 奖励发放：通过 RevenueCat Promotionals API 或等效方案自动发放，不依赖人工
- 进度可视化：用户随时可在产品内看到"距下次免单还差 X 天"
- 年付处理：积累额度，续费时统一抵扣

#### Paywall 设计规范
- 在用户最有意愿付费的时刻展示（完成首次体验后、Streak 里程碑、试用到期前）
- 必须展示 Free vs Pro 功能对比表，让用户清楚知道在买什么
- 一屏内完成付费（不要多步跳转）

### 4.2 AI 成本管控规范

所有使用 AI 能力的产品必须：

- **免费用户 vs 付费用户**：使用不同复杂度的 Prompt，控制 token 消耗差异
- **Rate Limit**：每用户每日 AI 调用次数上限（防止滥用）
- **缓存优先**：同一用户当日内容生成一次后缓存，不重复调用
- **相似复用**：相同偏好的用户内容可部分复用（成本优化）

## 五、安全规范

### 5.1 API Key 管理

| Key 类型 | 存储方案 | 绝对禁止 |
|---------|---------|---------|
| 系统级 API Key | Supabase Vault（pgsodium）/ AWS Secrets Manager | 明文存数据库、写入代码 |
| 用户自定义 Key | 设备本地 `flutter_secure_storage` / `keychain` | 上传服务器、存 DB 明文 |
| 第三方服务 Key | 环境变量（服务端）/ `.env`（不提交 git） | 硬编码在客户端代码中 |

### 5.2 客户端安全

- 永远不信任客户端传来的权限声明，服务端二次校验
- 敏感操作（支付、删除账号）必须二次确认
- 所有 API 请求走 HTTPS，禁止 HTTP
- 使用 Supabase RLS（行级安全）保护数据库，每张表都要配置策略

### 5.3 隐私规范

- 收集最小化原则：只收集产品功能必需的数据
- 用户数据可导出 + 可删除（GDPR 合规准备）
- 隐私政策在注册前可访问，不埋在深处

## 六、工程规范

### 6.1 Git 分支策略（简化 GitFlow）

```
main        — 生产分支，永远可发布，必须 PR + Review + CI
develop     — 集成分支，下一版本，必须 PR + CI
feature/*   — 功能分支，合入 develop，Squash Merge
fix/*       — 修复分支，合入 develop（或 main hotfix）
release/*   — 发版准备，合入 main + 回合 develop
hotfix/*    — 线上紧急修复，合入 main + 回合 develop
```

**分支命名**：`{type}/{issue-number}-{short-description}`
示例：`feature/42-streak-freeze-card`、`fix/87-tts-web-crash`

### 6.2 Commit Message 规范（Conventional Commits）

```
<type>(<scope>): <subject>

[optional body]
[optional footer: Closes #issue]
```

| Type | 用途 |
|------|------|
| `feat` | 新功能 |
| `fix` | Bug 修复 |
| `ui` | 纯 UI/样式改动 |
| `refactor` | 重构（不影响功能） |
| `perf` | 性能优化 |
| `test` | 测试 |
| `docs` | 文档 |
| `chore` | 依赖/配置 |
| `ci` | CI/CD |

**提交粒度**：每完成一个原子功能单元立即提交，不积压大 Commit。
一个页面 = 1 commit，一个 Service = 1 commit，一套主题配置 = 1 commit。

### 6.3 CI/CD 最低要求

每个项目必须配置以下自动化检查，PR 必须全部通过才可合并：

| 检查项 | Flutter | React/Web |
|--------|---------|-----------|
| 静态分析 | `flutter analyze --fatal-infos` | `eslint` / `tsc --noEmit` |
| 格式检查 | `dart format --check` | `prettier --check` |
| 单元测试 | `flutter test` | `jest` / `vitest` |
| 构建验证 | `flutter build web` | `npm run build` |

### 6.4 必须创建的工程文件

每个新项目必须在第一个 commit 中包含：

```
.gitignore                              — 平台标准忽略 + secrets 忽略
README.md                               — 项目说明 + 本地启动指引
CHANGELOG.md                            — Keep a Changelog 格式
Makefile                                — 一键式开发工作流（见 6.5）
.env.example                            — 环境变量示例（不含真实值）
.github/pull_request_template.md        — PR 描述模板
.github/workflows/ci.yml                — CI 自动化
.github/ISSUE_TEMPLATE/bug_report.md    — Bug 报告模板
.github/ISSUE_TEMPLATE/feature_request.md — 功能需求模板
```

### 6.5 Make 工作流规范（强制）

**所有项目必须提供 `Makefile`，作为开发者与项目交互的唯一入口。**

> 原则：新成员 clone 仓库后，只需 `make setup` + `make run` 即可完成本地启动，无需阅读额外文档。

#### 必须实现的 Make 目标

| 目标 | 说明 | 用途 |
|------|------|------|
| `make help` | 打印所有可用命令及说明 | 默认目标，无参数时执行 |
| `make setup` | 一键初始化：安装依赖、代码生成、复制 .env | 新成员第一次使用 |
| `make env` | 从 `.env.example` 复制 `.env`（已存在时跳过）| 环境配置 |
| `make gen` | 运行代码生成（Freezed / Riverpod / i18n）| 修改模型后 |
| `make run` | 启动开发服务器（Web 默认）| 日常开发 |
| `make check` | 本地运行 CI 全量检查（lint + format + test）| 提交前自检 |
| `make test` | 运行全部单元测试 | 日常测试 |
| `make lint` | 静态分析 | 单独检查 |
| `make format` | 自动格式化代码 | 保存后修复 |
| `make clean` | 清除构建产物和生成文件 | 排查问题 |
| `make build-web` | 生产构建（Web）| 部署前 |

#### 额外推荐目标（有后端服务时必须）

| 目标 | 说明 |
|------|------|
| `make supa-start` | 启动本地 Supabase 栈 |
| `make supa-stop` | 停止本地 Supabase |
| `make supa-reset` | 重置本地 DB（重新执行所有 migration）|
| `make supa-fn` | 本地热加载 Edge Function |
| `make deploy-fn` | 部署 Edge Function 到云端 |

#### Makefile 编写规范

```makefile
# ① 顶部声明所有 .PHONY 目标（避免与同名文件冲突）
.PHONY: help setup run test ...

# ② 默认目标必须是 help，无参数运行时显示帮助
.DEFAULT_GOAL := help

# ③ help 目标用 @echo 打印彩色分类说明
help:
    @echo "..."

# ④ 每个目标前加 guard，检查依赖工具是否已安装
_check-flutter:
    @which flutter > /dev/null 2>&1 || (echo "✗ flutter not found" && exit 1)

# ⑤ 输出风格统一：用颜色区分状态
#   ▶ 青色：正在执行   ✓ 绿色：成功   ✗ 红色：失败   ⚠ 黄色：警告
```

#### 维护要求

- 每次新增工具链命令（新的 CLI、新的服务），**同步更新 Makefile**
- `make setup` 必须在 CI 环境（无交互）中可执行（所有步骤不弹出确认）
- 每次修改 Makefile，同步更新 README 的"快速开始"章节

### 6.6 代码规范

- **注释原则**：只写"为什么"，不写"是什么"。代码能自解释的地方不加注释
- **函数长度**：单个函数不超过 50 行，超过则拆分
- **命名**：宁可名字长也要语义清晰，禁止缩写（除公认缩写如 `id`、`url`）
- **魔法数字**：禁止在代码中直接写数字/字符串常量，必须定义为命名常量

## 七、跨平台开发规范

### 7.1 平台支持优先级

所有 C 端产品默认支持三端：**iOS · Android · Web**

- **首选技术栈**：Flutter（一套代码，性能接近原生，UI 一致性最好）
- **替代方案**：Expo（React Native，适合已有 React 团队）

### 7.2 平台差异处理

| 功能 | iOS | Android | Web |
|------|-----|---------|-----|
| 登录 | Apple + Google | Google | Google / Email |
| 推送 | APNs | FCM | Web Push |
| 支付 | StoreKit 2 | Google Play Billing | Stripe |
| 本地存储 | Keychain | Keystore | localStorage / IndexedDB |
| TTS 语音 | AVSpeechSynthesizer | TextToSpeech | Web Speech API |

Web 端降级策略：原生能力不可用时，必须有 Web 兼容方案，不允许功能直接缺失。

### 7.3 响应式设计要求

- 移动端（< 768px）：单列布局，底部导航栏
- 平板（768px-1024px）：侧边栏 + 内容区
- 桌面（> 1024px）：固定最大宽度（推荐 480px 居中），保持移动端体验

## 八、需求转化规范

> 本节定义如何将用户/老板的口头需求转化为标准的产品规范条目。

### 8.1 需求接收 Checklist

收到任何新需求时，必须明确以下信息后才能进入 PRD：

- [ ] **用户是谁**：哪类用户有这个需求？（不是"所有用户"）
- [ ] **场景是什么**：用户在什么情境下需要这个功能？
- [ ] **现在怎么做**：用户没有这个功能时怎么解决问题？
- [ ] **成功标准**：做完了怎么判断是否成功？
- [ ] **优先级**：P0（阻塞上线）/ P1（本期必做）/ P2（有最好）/ P3（未来规划）

### 8.2 口头需求 → PRD 条目转化示例

| 口头描述 | 转化为规范条目 |
|---------|--------------|
| "要支持多语言" | 强制要求：所有文本必须走 i18n，Phase 1 支持 zh_CN + en_US，跟随系统语言，用户可覆盖，配置持久化 |
| "界面要好看高端" | 采用语义化色彩 Token，留白优先，衬线字体用于展示标题，微交互规范参考第二章，空状态必须设计 |
| "要能改颜色主题" | 预设 ≥ 4 套主题，每套含 Light/Dark，通过 ThemeExtension 热切换，主题选择持久化到用户配置 |
| "用户坚持了要有奖励" | 设计 Streak 里程碑系统，30/60/90/100 天节点，物质奖励（免单/冻结卡）+ 虚拟奖励（限定主题/徽章） |
| "要先做好设计再开发" | 强制研发流程：PRD → UI 设计稿 → TDD → 开发，任何阶段不得跳步，需求变更走版本控制 |

### 8.3 新需求评估框架

收到新需求后，用以下框架快速评估是否值得做：

```
Impact（影响）  × Confidence（把握）
─────────────────────────────────────  = 优先级分数
           Effort（工作量）
```

- Impact：对北极星指标的影响（1-10分）
- Confidence：我们有多大把握这个需求是真实的（1-10分）
- Effort：实现所需工作量（1-10，越大越难）

分数 > 5：优先考虑；分数 < 3：放入 Backlog
